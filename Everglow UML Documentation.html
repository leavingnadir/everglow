⚠ Circular Dependency Risk
DESIGN

BookingService calls PaymentService, and PaymentService calls BookingService to update status. This creates a circular dependency. Resolve by extracting a BookingStatusService or using an event-driven approach (Spring ApplicationEvent).

⚠ Vendor Denormalization in Payment
DATA

The original Payment entity stored vendorName : String — a denormalized copy. This is now removed; vendor info should be resolved via booking → vendor. Keeping a redundant copy risks data divergence.

⚠ Booking→Feedback Multiplicity
SCHEMA

Original diagram marked this as 1..1. A booking may involve multiple vendors (via a package), so a user should be able to leave separate feedback per vendor. Corrected to 1..* in this file.

⚠ Missing packageId FK on Booking
SCHEMA

The original class diagram had no link from Booking to Package. Without this FK the system cannot determine which package was booked, breaking package management and reporting. Added in both Class and ER diagrams.

ℹ Feedback Approval Flow
WORKFLOW

Feedback is saved with approved = false by default. Only an Admin can approve it via PATCH /feedback/{id}/approve. This admin-side action is not guarded in the original routes — ProtectedAdminRoute or a backend @PreAuthorize("ROLE_ADMIN") should protect it.

ℹ includedVendors — Separate Table
SCHEMA

Package.includedVendors : List<VendorCategory> is stored as an element collection. For complex querying, consider a package_vendor_categories join table instead of a serialized list.
